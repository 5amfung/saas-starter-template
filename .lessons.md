# Lessons Learned

## 2026-03-24: Read the library API contract before writing integration code

**Context:** Billing redesign — integrating Better Auth Stripe plugin with org-level billing.

**Mistake:** Wrote all billing server functions without reading the Stripe plugin source or docs for the `customerType` parameter. This caused:

1. Stripe customer created on user instead of organization
2. `organization.stripe_customer_id` stayed null
3. Added a hacky fallback to query `subscription.stripe_customer_id` instead of fixing root cause

**Rule:** Before writing any integration code with a third-party library:

1. Read the library's API contract (docs, source, types) for every endpoint you'll use.
2. Identify all required/optional parameters and their defaults.
3. When something doesn't work as expected, investigate the root cause in the library source FIRST — don't add workarounds.

## 2026-03-24: Fix root causes, not symptoms

**Context:** `organization.stripe_customer_id` was null after checkout.

**Mistake:** Instead of investigating why it was null, added a fallback to read from `subscription.stripe_customer_id`. This masked the real issue (`customerType: 'organization'` missing from API calls).

**Rule:** When data is unexpectedly null/missing, trace the write path to find where it should have been set. Don't add read-path workarounds.
